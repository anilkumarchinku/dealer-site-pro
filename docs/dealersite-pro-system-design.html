Multi-Tenant Dealer Website Platform

DealerSite Pro powers onboarding, dashboards, inventory, public dealer websites, leads, payments, reviews, domains, and deployment from one shared SaaS application.

Prepared for sharing: high-level system design, module breakdown, data model, flows, and technology stack.

1. System Purpose

DealerSite Pro is a multi-tenant SaaS platform for vehicle dealerships. Dealers can onboard, configure a website, add inventory, manage leads, connect domains, accept bookings or payments, and publish a public website without maintaining a separate codebase per dealer.

Dealer Signs UpSupabase Auth creates the user and links the dealer account.
OnboardingDealer profile, brands, services, templates, and inventory source are collected.
DashboardDealer manages stock, leads, model cards, webpage content, domains, and reviews.
Public WebsiteSlug or domain resolves to dealer config and renders selected templates.
Customer ActionsLeads, messages, test drives, bookings, and inquiries flow back to the dashboard.

3. Core Modules

Auth & Dealer Account

Supabase Auth manages users. Dealer records are linked to authenticated users and protected through ownership checks and RLS.

Onboarding

Collects vehicle category, dealership profile, brands, services, website style, images, domains, and inventory source.

Dashboard

Operational control center for inventory, model cards, leads, messages, reviews, analytics, webpage editing, and settings.

Public Dealer Website

Resolves dealer by slug or domain, fetches live configuration, and renders the selected template with inventory and lead CTAs.

Inventory

Supports manual add, bulk CSV upload, uploaded images, Cyepro inventory sync, and catalog-backed new vehicle pages.

Domains & Deployment

Supports free subdomains, custom domains, DNS verification, Vercel domain registration, and Cloudflare automation.

4. Main Request Flows

Dashboard Save Flow

  • Dealer opens dashboard page.
  • Next.js loads state and fetches dealer data from Supabase.
  • User edits inventory, website design, model cards, or settings.
  • API/client write validates dealer ownership.
  • Supabase updates data and public pages render latest data.

Public Website Flow

  • Customer visits slug or custom domain.
  • App resolves dealer and site configuration.
  • Inventory comes from manual stock, Cyepro, or static catalog.
  • Selected template renders the public website.
  • Lead, call, WhatsApp, test drive, and booking actions are tracked.

7. Security, Reliability, and Operations

Security Controls

  • Supabase Auth for dealer login.
  • Dealer ownership validation before sensitive writes.
  • Supabase RLS policies for table access.
  • Razorpay webhook signature verification.
  • Content Security Policy and security headers in Next.js config.

Reliability Controls

  • Sentry for server and client error monitoring.
  • Upstash rate limiting for sensitive APIs.
  • Idempotency for payment and webhook flows.
  • Vercel deployment history and rollback support via Git/GitHub.
  • Dynamic rendering for public dealer pages that need fresh inventory.

8. Deployment Architecture

Local DevelopmentCode changes are made in the project workspace.
GitChanges are tracked with commits and branches.
GitHubRepository stores source, PR history, and collaboration workflow.
VercelBuilds and deploys the Next.js application.
ProductionDealers and customers access dashboard and websites through live URLs.

Supabase remains the shared backend across deployments, while public sites are resolved dynamically by slug or domain.

9. Summary

The system is designed as a single multi-tenant SaaS application. The same Next.js codebase powers onboarding, dealer operations, and public dealer websites. Supabase is the source of truth for identity, data, and assets. External services such as Razorpay, Vercel, Cloudflare, Cyepro, Sentry, Upstash, Git, and GitHub provide payments, deployment, domains, inventory integration, monitoring, protection, and engineering workflow.
